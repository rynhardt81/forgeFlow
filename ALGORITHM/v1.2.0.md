# Forge Flow Algorithm v1.2.0

> Doctrine for non-trivial tasks. Read this file at the start of an Algorithm-mode task and follow it through all phases. It is a verification harness, not a ceremony: every rule here exists to stop unverified work from being called done.

## v1.1.0 → v1.2.0 diff

- **Removed all output ceremony**: emoji phase banners, the mandatory final summary format, and fixed-word-count scaffolds. Telemetry showed they aggravated the failure modes this doctrine exists to prevent (premature completion claims, truncated answers). The *checks* survive as prose; the formats do not.
- **Removed ISC count floors and thinking floors.** Criterion count is judgment, gated by the splitting test and the quality gates below — not a quota. (Calibration: the entire v3 framework rebuild needed 33 ISCs; no 45-minute task needs 32.)
- **Removed time budgets from effort tiers** — tiers describe scope, not stopwatch targets.
- **Escalation default inverted**: Algorithm mode fires on explicit signals (see CLAUDE.md Modes), not "anything else". Over-escalation was the observed failure, not under-escalation.
- **Folded in phased hardening** (formerly `rules/phased-hardening.md`) under Anti-criteria.
- Unchanged: phase order, reproduce-first, inline verification mandate, live-probe rule, re-read gate, Background Agent Checkpoint Discipline, plan-means-stop, Tier-2 authority.

## Doctrine — internalize before starting

**Every Algorithm run does one thing: transition from CURRENT STATE to IDEAL STATE.** Articulate the ideal state as testable criteria (ISCs), pursue them through phases, verify each one met. Same primitive for code, debugging, refactoring, design.

**The ISA is the system of record.** One document holding (1) the ideal-state articulation, (2) the test surface (ISCs), (3) the verification trail, (4) the done condition. Either `<project>/ISA.md` (long-lived project work) or `docs/tasks/<task-id>/ISA.md` (per-task). Created at OBSERVE, refined through pursuit, verified at VERIFY.

**ISCs are atomic.** Each criterion describes one verifiable end-state with a single nameable tool probe. Split until that's true. If a criterion needs human judgment, name the tool-verifiable proxy.

**Anti-criteria are required.** Every ISA has ≥1 ISC of the form "Anti: <what must NOT happen>". They make out-of-scope and regression-prevention probe-able. **Phased hardening rides here:** the current task ships the core `Anti:` criteria; deeper hardening (rate limits, abuse cases, load) becomes follow-up tasks filed via `forge task add` — never padding on the current ISA, never silently skipped.

**Verification is real probes, not assertions.** Mark `[x]` only with tool-verified evidence in the ISA `## Verification` section.

**Long-running background work is checkpointed work.** In a spawned git worktree, commit progress periodically. Default is checkpoint-on. See Background Agent Checkpoint Discipline.

## Effort tiers

| Tier | Scope | ISA |
|------|-------|-----|
| **E1** | Trivial, fast-path | none |
| **E2** | Single-domain substantial work | inline criteria checklist in the reply |
| **E3** | Multi-file, requires planning | ISA document required |
| **E4** | Architectural, cross-cutting | ISA document required |

User can override with `/e1`–`/e4`. Criterion count at any tier is judgment: enough that every deliverable, risk, and regression path has a probe — no more.

## Phases

Fixed order: OBSERVE → THINK → PLAN → BUILD → EXECUTE → VERIFY → LEARN. State plainly which phase you are in when it changes; no banner format required.

### OBSERVE

**First action:** restate the user's request in one sentence — explicit asks, implied out-of-scope, urgency. If you can't restate it accurately, re-read the message.

**Preflight gates** — fire any that match:

| Gate | Trigger | Goal |
|------|---------|------|
| **A: Diagnostic** | Bug-fix, "X broken", debugging | Reproduce failure before reading code |
| **B: External service** | API, DB, infrastructure | Confirm credentials and access exist |
| **C: Research** | Unfamiliar library, error messages | Search docs before code archaeology |
| **D: Background-agent context** | Running inside a spawned worktree | Enter Checkpoint Discipline; commit `wip(checkpoint): observe start` before any tool action |
| **E: Schema change** | Migration files, DDL, model/schema edits in scope | Read `rules/migrations.md`; backup-verified and rollback-paired before apply |

**REPRODUCE-FIRST (Gate A):** a reproduction must be captured before any Read/Grep targets the suspect code path.

| Symptom | Required reproduction |
|---------|----------------------|
| Web/UI bug | Browser screenshot or network trace |
| HTTP failure | `curl -i` showing the broken response |
| CLI failure | Actual stdout/stderr |
| Test failure | The failing test output |
| Data inconsistency | `SELECT` showing the wrong value |

**Project context check:** before scaffolding the ISA, read the relevant Tier 2 source-of-truth docs from `.claude/reference/` — `02` for architecture-affecting work, `03` + `08` for security-relevant work, `05` for deployment work. The ISA `## Constraints` references these as authoritative.

**Optional — vet the idea first (`/vet-idea`):** for a new idea/decision of uncertain worth, the user may run `/vet-idea` before any ISA exists. NO-GO halts the run; GO's named risks seed `Anti:` ISCs. Manual-invoke only.

**Scaffold the ISA** (E3+; inline checklist at E2). **Optional — ISA Interview (E3/E4):** when unresolved design decisions depend on each other, run `skills/ISA/Workflows/Interview.md` before freezing ISCs.

**Build ISCs.** Apply the splitting test to every criterion:

| Test | Split when |
|------|-----------|
| "And"/"With" | Joins two verifiable things |
| Independent failure | A can pass while B fails |
| Scope words | "all", "every", "complete" → enumerate |
| Domain boundary | Crosses UI / API / data → one per boundary |
| **No nameable probe** | You can't say which tool verifies it |

**ISC quality gates** — all must pass before THINK: every ISC has a nameable single-tool probe; every deliverable maps to ≥1 ISC; ≥1 `Anti:` ISC present.

### THINK

Name the riskiest assumptions the work depends on, the premortem failure modes it must withstand, and any blockers. Re-apply the splitting test; add criteria for premortem failure modes.

**Stress-test the load-bearing assumptions (E3/E4):** take the top 1–2 riskiest assumptions: (1) restate each as a measurable observable, (2) name the counter-evidence you'd expect if it's false, (3) check whether the work still ships if it's materially wrong, (4) for the assumption whose failure hurts most, name a validation step + contingency + early-warning trigger. Wire outcomes into the existing rails: survival outcome → a THINK-derived ISC; early-warning trigger → an `Anti:` ISC; threshold → the ISA Test Strategy. Skip at E1/E2.

### PLAN

State scope (depth / breadth / breadth-then-depth) and whether the root cause is identified or still needs investigation. If the request has 2+ explicit sub-tasks, list them as deliverables — each mapping to ≥1 ISC — quoting the user's distinctive phrasing so nothing silently drops.

**Plan means stop** — if the user asked for a plan, present it and STOP. No execution without approval.

### BUILD

Prepare: read targets, set up the environment, confirm tooling.

**Root-cause-at-ingestion checkpoint** (for any fix that modifies output-side behavior):
1. Where does the bad state enter the system?
2. If I fix at the ingestion point instead, do similar bugs disappear?
3. For UI bugs, am I tracing display-down (correct) or database-up (likely wrong)?

**Background-agent checkpoint** (Gate D): commit `wip(checkpoint): build start — <targets read>` before execution.

### EXECUTE

Do the work. As each ISC passes, edit the ISA: `[ ]` → `[x]`, append a `## Verification` entry.

**Inline verification mandate** — no `[ ]` → `[x]` transition without evidence in the same tool block:

| ISC type | Minimum verification |
|----------|---------------------|
| File write | `Read` confirming content |
| Code edit | `Grep` for the new symbol or `Read` the range |
| Command | `Bash` with checked output |
| HTTP | `curl -i` with status + body shape |
| UI | Browser screenshot at the target route |
| Schema | `SELECT` confirming migration |

**Forbidden language**: "should work", "should be", "expected to", "the change is in place" (without Read/Grep), "no errors" (without the actual log).

**Atomic task-state mutation:** all task state changes go through `python3 .claude/scripts/forge/forge.py task ...`. Never hand-edit `docs/tasks/registry.json` — the `consistency-banner.py` hook surfaces drift on every write.

### VERIFY

**Live-probe rule:** if an ISC covers a user-facing surface (web page, HTTP endpoint, CLI output, DB write, deploy), `[x]` requires tool-verified probe evidence — not "tests pass" or "looks fine". If a live probe is genuinely impossible at execution time, mark `[DEFERRED-VERIFY]` with a follow-up task ID.

**Validator firing:** agent-bound validators fire on PostToolUse output writes; skill-bound validators fire per their frontmatter (mostly at Stop). Their `additionalContext` advice is part of the evidence trail — read it, don't discard it.

**Re-read gate** — final check before LEARN: re-read the user's last message verbatim, enumerate every ask, and confirm each is addressed. Any miss blocks `phase: complete`.

**Background-agent final commit** (Gate D): before signalling completion, `git status --porcelain` must be empty on the worktree branch; residual state is promoted via `wip(reaped): final state at completion`.

### LEARN

Ask: what should have been done differently, what did verification catch, did the preflight gates earn their firing? Route anything worth keeping:

| TYPE | Target | Action |
|------|--------|--------|
| `knowledge` | `docs/project-memory/` (via `/remember`) | Inline write |
| `rule` | Project `CLAUDE.md` | Inline append |
| `gotcha` | Skill SKILL.md Gotchas section | Inline append |
| `doctrine` | Algorithm `v<next>.md` (proposal) | Surface to user |
| `architecture` | `reference/02-architecture-and-tech-stack.md` | Inline append |
| `decision` | `reference/06-architecture-decisions.md` | Inline append (ADR format) |

Default disposition: SKIP. Most session insights aren't worth persisting.

Set `phase: complete` on the ISA. (The `isa-reflection-emit.py` hook appends a reflection skeleton to `daily/algorithm-reflections.jsonl` automatically at E2+.)

## Background Agent Checkpoint Discipline

**Applies when:** a background agent operates inside a worktree spawned via `isolation: "worktree"` with `run_in_background: true`, OR `WORKTREE_AGENT_ID` is set, OR the current branch starts with `wip-`. Gate D fires on these.

**Why:** background agents have no commit safety net. If the worktree is reaped before the agent commits, the work is lost — and the git history is the only truthful record of work.

1. **Periodic checkpoint commits** — every ~5 minutes of activity or ~10 meaningful tool actions, whichever first. Message starts `wip:` or `wip(checkpoint):` (autosquash-friendly). Scratch notes and partial diffs qualify.
2. **Checkpoint-before-stuck-operation** — before any operation likely to hang (long test runs, long commands, slow curl probes): `wip(checkpoint): before <operation> — <reason>`.
3. **Parent-side heartbeat read** — the parent reads progress via `python3 .claude/scripts/forge/forge.py agent heartbeat <branch> [--json]`, never the agent's JSONL transcript (context overflow). Heartbeat over commit log is the contract.
4. **No-commit-no-reap** — Forge-owned cleanup refuses to delete a worktree with a dirty `git status --porcelain`; forced cleanup first promotes state to `wip(reaped): final state at cleanup` and prints the SHA. See `scripts/forge/worktree_safe_cleanup.py`.

Squash conventions: `wip:` prefixes are reserved; final PR squash collapses them via `git rebase -i --autosquash`; the final non-wip commit describes the deliverable. Skill authors may flag `--no-checkpoint` for known-short-lived agents; default is checkpoint-on. Not required for foreground work, read-only agents, or single-action agents. If in doubt, checkpoint.

## Rules

- **No phantom capabilities** — anything you claim to have done must correspond to a real tool call.
- **ISA is yours to write** — no hook generates it; you edit via Read/Edit/Write.
- **ISC IDs are stable** — never renumber on edit; splits become `ISC-N.M`, drops become tombstones.
- **The ISA is the test harness** — for project ISAs, ISCs cover application logic, security, build, deploy.
- **Plan means stop** — present and STOP; no execution without approval.
- **Tier 2 docs are authoritative** — when ISA constraints conflict with `reference/02`, the reference doc wins until reconciled via ADR.
- **Background agents checkpoint by default** — opt-out is explicit (`--no-checkpoint`), opt-in is implicit.
